并发问题梳理及解决方案设计

一、什么是并发问题

并发问题的核心是：多个执行单元同时访问和修改同一份数据时，因执行顺序、时机或协作不当而导致的数据错误、业务逻辑异常或系统性能下降。

1.1 常见问题表现

问题类型 简单定义 业务表现
数据竞争 多个执行单元同时写入同一数据，没有同步保护 库存扣成负数、重复发券、重复扣款
死锁 多个执行单元互相持有对方需要的资源，互不释放 接口长时间无响应
状态不一致 并发操作导致关联数据在不同地方不一致 数据库已更新但缓存未刷新
竞态条件 程序行为依赖于不可控的执行顺序 结果不稳定、偶现问题

二、并发场景分类（MECE）

2.1 按场景类型

分类 说明
同一业务对象被同时修改 同一订单、同一报名记录、同一库存单元
同一业务动作被重复触发 重试、超时补偿、重复提交、消息回放
跨链路状态需要一致但更新节奏不同 A 已成功，B 仍旧值
同一业务结果由多条链路共同产出 在线链路、离线任务、人工操作并行
资源容量竞争导致并发退化 连接池、锁等待、队列堆积

2.2 按执行单元

分类 说明
单一执行单元内多线程 同一请求内部开多线程处理共享数据
多个执行单元并发 不同请求、定时任务、消费者同时修改同一数据

三、并发问题定位方法

3.1 四层检查模型

  1. 业务对象层：定义”谁是共享对象”，明确唯一标识与边界
  2. 业务动作层：盘点所有会改变对象状态的动作（接口、任务、消费者、人工操作）
  3. 时序层：识别哪些动作会在时间上重叠，特别是”先读后写”窗口
  4. 一致性层：明确该对象允许的最终状态、临时状态和不可逆状态

3.2 逐步排查流程

第一步：列清楚所有写入口

第二步：检查各写入口是否有并发保护

  • ❌ 没有任何保护（最危险）
  • ✅ 有分布式锁
  • ✅ 有数据库级别条件检查（版本号/状态）
  • ✅ 有状态机约束
  • ✅ 有数据库行锁（SELECT FOR UPDATE）

第三步：检查保护机制是否合理

  • 锁粒度是否和实际修改数据维度一致？
  • 加锁失败后代码是否禁止写操作？
  • 版本号是否在数据库层原子性更新？

第四步：检查多个写入口之间是否协调

四、并发保护方案

4.1 分布式锁（悲观锁）

适用场景：冲突频率高、冲突代价高、业务逻辑复杂

关键要点：

  • 超时时间设为业务 TP99 的 1.5～2 倍
  • 加锁失败后严禁继续执行写操作
  • 选择合适的锁粒度（锁维度要和数据维度一致）

五、方案选择速查

场景 推荐方案
错误一次就不可接受（金钱、库存） 分布式锁 + 状态机
高并发且可重试 乐观锁 + 幂等
跨系统多写点 主事实约束 + 最终一致性编排
存在人工与系统并行修改 统一业务键 + 统一状态迁移规则

六、常见陷阱

陷阱 1：加锁失败后仍然执行写操作

陷阱 2：锁超时时间太短

陷阱 3：版本号在应用层计算

陷阱 4：加锁后未重新查询数据

陷阱 5：遗漏定时任务或异步消费者

排查写入口时，不要只关注 HTTP 接口，还要包括：

  • Crane/XXL-Job 定时任务
  • MQ 消息消费者
  • 异步线程池任务
  • 人工触发的批量脚本

七、并发保护检查清单

对每个修改数据的入口，确认以下项目：

  • 这个入口修改的是什么数据？（具体到表、字段、业务维度）
  • 有没有其他入口也修改同一数据？
  • 是否有并发保护机制？
  • 保护粒度是否和实际修改数据的维度一致？
  • 如果是分布式锁，超时时间是否合理？加锁失败后是否禁止写？
  • 如果是乐观锁，版本号是否在数据库层原子性更新？
  • 加锁后是否重新查询了最新数据？
  • 是否遗漏了定时任务/异步任务的并发场景？
